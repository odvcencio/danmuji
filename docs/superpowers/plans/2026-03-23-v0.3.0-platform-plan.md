# Danmuji v0.3.0 Platform Plan

> **Goal:** Make danmuji the default way to delete repetitive Go test setup, assertions, doubles, fixtures, and harness code without trapping users away from raw Go.

> **Release policy:** Ship all scope below in a single public `v0.3.0`. Internally, implement in slices and maintain a green branch after every slice.

---

## Product Goal

Danmuji should remove boilerplate from the six places Go tests usually get noisy:

1. Data construction
2. Assertions and failure diffs
3. Environment and fixture setup
4. Test doubles
5. Async/concurrency waiting patterns
6. Migration from existing handwritten Go tests

The core rule is unchanged: **danmuji is Go plus targeted test superpowers**. Raw Go remains the escape hatch inside every block.

---

## Scope

`v0.3.0` includes:

1. `factory` / `build`
2. rich assertions: `matches`, `unordered_equal`, `err is`, `err message contains`
3. richer `needs` fixtures, starting with `tempdir` and `http`
4. interface-derived `mock` / `fake`
5. `await <-ch within ... as ...`
6. `danmuji convert` for common Go test shapes

`v0.3.0` does **not** include:

- a new non-Go programming model
- many more global keywords than needed
- full-featured arbitrary code conversion
- a custom matcher language that cannot fall back to normal Go expressions

## Real-World Blockers

The following issues came directly from migrating real goetrope tests and must be treated as `v0.3.0` blockers, not stretch goals:

1. Struct literal parsing inside danmuji blocks
2. `needs` env vars plus lifted test-function scope
3. WebSocket helper parity with `danmujiHTTP`
4. Matrix variable scoping across nested `given` / `when` / `then`
5. Unused import injection cleanup
6. Pre-built binary support for `process run ...` combined with lifted `needs` scope
7. `eventually` with typed assertions on async helpers
8. gRPC streaming helper parity for protocol tests

Priority order for maximum impact:

1. struct literal parsing
2. `needs` env vars and lifted scope
3. WebSocket helper
4. matrix variable scoping
5. unused import cleanup

## Syntax Spec

### 1. Factories

Top-level declaration:

```dmj
factory User {
	defaults {
		name: "test-user"
		role: "member"
		email: fake.email()
	}

	trait admin {
		role: "admin"
	}

	trait suspended {
		state: "suspended"
	}
}
```

Build expressions:

```dmj
user := build User
admin := build User with admin
named := build User with admin { name: "alice" }
```

Rules:

- `factory` is top-level only
- `defaults` is required inside a factory body
- `trait` is optional and may appear multiple times
- `build Name` creates a fresh value
- `with trait_name` applies one or more traits in order
- `{ ... }` at the end applies last-mile overrides
- values remain plain Go expressions

MVP semantics:

- generated factory helpers return `map[string]interface{}`-style field sets internally
- emitted `build` code materializes a struct literal when the destination type is obvious, otherwise returns a `map[string]interface{}` helper result
- fake-data helpers are deterministic by default

Implementation note:

- Reuse the existing `scenario_field` extraction patterns from [transpile_data.go](/home/draco/work/danmuji/transpile_data.go)

### 2. Rich Assertions

Supported forms:

```dmj
expect user matches {
	role: "admin"
	email: contains "@"
	deleted_at: is_nil
}

expect items unordered_equal expected
expect err is context.DeadlineExceeded
expect err message contains "quota exceeded"
```

Rules:

- `matches` performs partial matching for structs and maps
- `contains`, `is_nil`, `not_nil` are allowed as field-level match operators
- `unordered_equal` compares slice/set content independent of order
- `err is expr` uses `errors.Is`
- `err message contains expr` compares `err.Error()`

MVP semantics:

- matching emits diff-rich helpers with field paths
- `unordered_equal` works for slices of comparable values and JSON-ish shapes
- assertions should reuse the existing `expectFailureContext` flow in [transpile_core.go](/home/draco/work/danmuji/transpile_core.go)

### 3. Fixtures

Supported forms:

```dmj
needs tempdir fs

needs http server {
	handler api.NewHandler(repo)
}

needs postgres db {
	migrate "./migrations"
	seed "./testdata/users.sql"
}
```

Rules:

- `needs tempdir name` transpiles to `t.TempDir()`
- `needs http name` spins up an `httptest.Server`
- `handler <expr>` is required for `http`
- `postgres` keeps the current container-backed shape and gains optional `migrate` / `seed`

MVP semantics:

- every fixture cleans itself up automatically
- HTTP fixtures integrate with the existing `danmujiHTTP` helper family in [transpile_fixtures.go](/home/draco/work/danmuji/transpile_fixtures.go)
- richer `needs` continues to compile to readable Go rather than runtime magic

### 4. Interface-Derived Doubles

Supported forms:

```dmj
mock UserRepo from repository.UserRepo

fake Mailer from mail.Mailer {
	Send(msg Message) -> error {
		return nil
	}
}
```

Rules:

- `from` is optional for old handwritten danmuji mocks/fakes, required for auto-derived ones
- interface source may be local or imported
- auto-derived mocks generate call recording and defaults
- auto-derived fakes generate stubs that must be overridden or panic clearly

MVP semantics:

- introspect interfaces using `go/packages` rather than tree-sitter-only parsing
- preserve current `mock` / `fake` behavior in [transpile_profile_doubles.go](/home/draco/work/danmuji/transpile_profile_doubles.go)
- extend `verify ... called with ...` to use generated call capture from interface-derived mocks

### 5. Async / Concurrency

Supported forms:

```dmj
await <-jobs within 2s as job
expect job.ID == "123"

eventually "job completes" within 5s {
	expect repo.Status(id) == "done"
}
```

Rules:

- `await <-expr within duration as identifier`
- timeout is required
- failure message includes channel expression, timeout, and enclosing test context
- `eventually` / `consistently` keep raw Go bodies and gain better timeout diagnostics

MVP semantics:

- `await` transpiles to `select` + `time.After`
- initial version supports single-value channel receives only
- a later extension may add `await value, ok := <-ch`, but not in the first implementation slice

### 6. Conversion Tooling

CLI:

```bash
danmuji convert ./mypkg
danmuji convert ./mypkg --write
danmuji convert ./mypkg/user_test.go --stdout
```

Initial conversion targets:

1. `assert.Equal`, `assert.NotEqual`, `require.NoError`, `assert.Error`
2. `httptest.NewRequest`, `httptest.NewRecorder`
3. table tests using `[]struct{...}` and `t.Run`
4. simple handwritten fake/mock structs when the pattern is obvious

Rules:

- conversion is best-effort and explicit
- unsupported patterns remain as raw Go inside generated danmuji blocks
- conversion must never silently change behavior

## Risk Register

### 1. Grammar ambiguity

Risk:
- too many bare keywords create collisions with normal Go

Mitigation:

- prefer contextual syntax (`build User`, `mock X from Y`, `await <-ch within 2s as x`)
- add soft keywords only when real collisions appear
- keep regression tests for identifier reuse

### 2. Interface loading complexity

Risk:
- imported interface resolution gets brittle

Mitigation:

- use `go/packages` instead of custom parsing
- cache interface metadata by package/type name
- fail with explicit diagnostics when an interface cannot be resolved

### 3. Conversion quality

Risk:
- converter becomes unreliable and damages trust

Mitigation:

- limit `v0.3.0` converter scope to common patterns only
- emit warnings and raw Go fallback for unknown constructs
- use golden tests for every supported pattern

### 4. Runtime helper sprawl

Risk:
- helpers become inconsistent and hard to debug

Mitigation:

- centralize helper emission in [transpile_fixtures.go](/home/draco/work/danmuji/transpile_fixtures.go)
- keep helper names stable and readable
- prefer a few shared helpers over many tiny one-off helpers

### 5. Parse recovery blind spots

Risk:
- typed expressions inside data blocks can knock the parser off the intended construct and produce misleading outer-block errors

Mitigation:

- add regression coverage for cast-heavy scenario/default expressions
- improve overlay selection so inner `given` / `then` / `each` sites win over outer test-block summaries
- keep recovery examples ergonomic enough that users do not have to bisect valid files by hand

Implement first:

1. grammar for `factory` / `build`
2. transpiler support for factories
3. parser and transpiler tests

Reason:

- highest immediate boilerplate reduction
- smallest risk of destabilizing the rest of the grammar
